В игре есть четкая иерархия объектов и можно выстроить четкую логику взаимодействия между ними.
- Абстрактные классы и наследование:
Базовый абстрактный класс для всех элементов игрового поля. Он определяет общие свойства, те координаты плит, их освещенность. Наследниками будут пустые клетки, которые можно осветить, стена/черный блок, который может содержать цифру/часть картинки и лампу которую ставит юзер.
- Инкапсуляция:
Будет использ-ся чтобы скрыть внутреннюю логику проверки светит ли лампа на другую или нет (внутри класса). Можно использовать приватные атрибуты (ex: grid),  чтобы состояние поля нельзя было бы изменять через сам интерфейс игры напрямую без каких-либо проверок .
- Полиморфизм:
Метод команды для отрисовки объекта (нарисуй что-то в координатах х,у) и метод визуализации, чтобы не просто нарисовать линию, а создать целую сцену, они могут быть определены в базовом классе, но каждый наследник будет использоваться, то есть отрисовываться по-разному (ex: стена черная, лампа желтая)
- Специальные методы: 
 1) __init__: инициализация игрового поля, установка размера и сложности
 2) __str__: позволит вывести игровое поле в консоль в виде текстовых символов
 3) __getitem__/__setitem__/__delitem__: сделает класс центра игры (ктр с инкапсуляцией, знает все о правилах игры, расположении объектов, о наступлении победы) очень удобным, тк можно будет обращаться к определенной плитке/клетку как board[x,y] = Lamp(), а __delitem__ будет убирать лампу с поля
 4) сравнения  __lt__, __eq__ и тд: можно использовать для сравнения рекордов игрока (скорости времени прохождения) или для сравнения сложности разных уровней на основе количества плит/клеток и лампочек.

# Полный разбор игры Akari

## Файл 1: `corepl.py` — ядро игры

### Строки 1–16: Шапка и импорты

```python
from abc import ABC, abstractmethod
```

`abc` — это встроенный модуль Python (Abstract Base Classes). `ABC` — специальный класс, от которого нужно наследоваться, чтобы твой класс стал **абстрактным**. Абстрактный класс — это класс-шаблон, который нельзя создать напрямую (`AbstractGameObject()` выбросит ошибку), он только задаёт интерфейс. `abstractmethod` — декоратор, которым помечают методы, которые обязан реализовать каждый потомок.

```python
from enum import Enum
```

`Enum` — перечисление. Вместо того чтобы писать магические строки вроде `"dark"` или `"lit"` по всему коду, мы создаём тип с фиксированным набором значений. Если опечататься — Python выдаст ошибку сразу, а не при запуске.

```python
from typing import Optional, Iterator, List, Tuple
```

Это **аннотации типов** — подсказки для разработчика и IDE, что именно хранится в переменной. `Optional[float]` значит «либо `float`, либо `None`». Сам Python их не проверяет в рантайме, но они делают код читаемым и помогают поймать ошибки заранее.

### `CellState` — состояния клетки (строки 28–42)

```python
class CellState(Enum):
    DARK     = "dark"
    LIT      = "lit"
    BULB     = "bulb"
    CONFLICT = "conflict"
```

Каждое имя — это константа. `CellState.DARK` — не просто строка `"dark"`, а объект типа `CellState`. Это важно: можно писать `if cell.state == CellState.LIT:` вместо `if cell.state == "lit":`, и не бояться опечаток.

### `Difficulty` — сложность (строки 44–57)

```python
class Difficulty(Enum):
    EASY = 1
    ...
    def grid_size(self) -> int:
        sizes = {1: 5, 2: 7, 3: 10, 4: 12, 5: 15}
```

`Enum` может иметь **методы**. `self.value` — числовое значение константы (для `EASY` это `1`). Метод `grid_size()`превращает уровень сложности в размер поля: `Difficulty.HARD.grid_size()` вернёт `10`.

### `AbstractGameObject` — корень иерархии (строки 62–90)

```python
class AbstractGameObject(ABC):
```

**Теория ООП: абстрактный класс.** Этот класс наследует `ABC`, значит он абстрактный. Его задача — гарантировать, что у каждого игрового объекта будут методы `__str__` и `__repr__`.

```python
    @abstractmethod
    def __str__(self) -> str:
        pass

    @abstractmethod
    def __repr__(self) -> str:
        pass
```

`@abstractmethod` означает: «каждый конкретный подкласс обязан написать свою версию этого метода». Если подкласс забудет — Python выбросит `TypeError` при попытке создать объект.

```python
    def __eq__(self, other: object) -> bool:
        if not isinstance(other, self.__class__):
            return NotImplemented
        return True
```

`__eq__` — **магический метод**, который вызывается при `==`. `isinstance(other, self.__class__)` проверяет, что мы сравниваем объекты одного класса. `return NotImplemented` — специальное значение, говорящее Python «я не умею сравнивать это, попробуй с другой стороны».

```python

        return self.__class__.__name__
```

`self.__class__` — это сам класс объекта. `.__name__` — его имя строкой. Для объекта `BulbCell` вернёт `"BulbCell"`. Это обычный (не абстрактный) метод — он уже реализован и доступен всем потомкам бесплатно.

### `AbstractCell` — абстрактная клетка (строки 96–149)

```python
class AbstractCell(AbstractGameObject):
    def __init__(self, row: int, col: int):
        self.__row = row
        self.__col = col
```

**Теория ООП: инкапсуляция.** Двойное подчёркивание `__row` — это **name mangling** (искажение имени). Python автоматически переименует этот атрибут в `_AbstractCell__row`. Это означает, что извне класса напрямую к нему не добраться: `cell.__row` выбросит ошибку. Данные скрыты — это и есть инкапсуляция.

```python
    @property
    def row(self) -> int:
        return self.__row
```

`@property` превращает метод в **свойство**. Теперь можно писать `cell.row` без скобок, и это вызовет метод под капотом. Это классика инкапсуляции: данные скрыты (`__row`), но доступны через контролируемый интерфейс (`row`).

```python
    @property
    def position(self) -> Tuple[int, int]:
        return (self.__row, self.__col)
```

Возвращает кортеж — пару координат. Используется везде, где нужно передать позицию клетки как единое значение.

```python
    @abstractmethod
    def is_passable(self) -> bool:
        pass
```

**Полиморфизм.** Каждый вид клетки по-своему отвечает на вопрос «проходит ли через тебя свет?». Стена — `False`, пустая клетка — `True`. Один и тот же вызов `cell.is_passable()` даёт разный результат в зависимости от типа клетки.

```python
    def __hash__(self) -> int:
        return hash((self.__row, self.__col, type(self)))
```

`__hash__` нужен, чтобы объект можно было класть в `set` или использовать как ключ `dict`. `hash()` от кортежа — стандартный способ создать хэш из нескольких полей. Включаем `type(self)`, чтобы `EmptyCell(0,0)` и `WallCell(0,0)` имели разные хэши.

### `EmptyCell` — пустая клетка (строки 155–225)

`super().__init__(row, col)` — вызов конструктора родителя (`AbstractCell`). ```python

```python
            raise ValueError(...)
```

`@state.setter` — это **сеттер**. Когда пишешь `cell.state = CellState.LIT`, Python вызывает этот метод. Внутри — валидация: нельзя поставить некорректное значение. Это защита данных через инкапсуляцию.

```python
    def add_light_source(self, bulb_pos: Tuple[int, int]) -> None:
        if bulb_pos not in self.__lit_by:
```

Добавляет лампочку в список источников света. Проверяет через `not in`, чтобы не добавлять одну лампочку дважды. После этого клетка всегда становится `LIT`.

```python
        if bulb_pos in self.__lit_by:
            self.__lit_by.remove(bulb_pos)

            self.__state = CellState.DARK
        else:
            self.__state = CellState.LIT
```

Убирает лампочку из списка. Если источников не осталось — клетка гаснет. Если остались другие — остаётся освещённой. Это корректная обработка пересечения лучей.

### `Grid` — сетка (строки 313–407)


Двойной **list comprehension**: создаём список из `size` строк, каждая — список из `size` столбцов, каждый элемент — `EmptyCell`. `__cells` скрыт через двойное подчёркивание.

```python

        row, col = pos

        return self.__cells[row][col]
```

`__getitem__` — магический метод для квадратных скобок. Позволяет писать `grid[2, 3]` вместо `grid.get_cell(2, 3)`. `row, col = pos` — распаковка кортежа.

```python

        row, col = pos
        self.__validate_pos(row, col)
        if not isinstance(cell, AbstractCell):
            raise TypeError(...)

```

`__setitem__` — для записи: `grid[2, 3] = WallCell(2, 3)`. Проверяет, что кладём только `AbstractCell` — это защита типов.

```python

            for cell in row:
                yield cell
```

`__iter__` — магический метод для итерации. `yield` делает из метода **генератор**: не создаёт весь список сразу, а отдаёт клетки по одной. Позволяет писать `for cell in grid:` — это обходит все клетки слева-направо, сверху-вниз.

```python

        return 0 <= row < self.__size and 0 <= col < self.__size
```

`__contains__` — для оператора `in`: `if (5, 3) in grid`. Используется везде в логике лучей, чтобы не выйти за пределы поля.


Приватный метод (одно подчёркивание — соглашение, что это внутренний метод). Вынесена общая проверка координат, чтобы не дублировать её в `__getitem__` и `__setitem__`.

Размер клетки вычисляется динамически: `560 // size` — поле 560px делим на число клеток. `min(62, ...)` — не больше 62px. `max(34, ...)` — не меньше 34px. Так поле всегда помещается на экран.

### `main()` — точка входа

```python
def main():
    app = QApplication(sys.argv)
    app.setStyle("Fusion")
    palette = QPalette()
    palette.setColor(QPalette.ColorRole.Window, QColor(14, 15, 20))
    app.setPalette(palette)
    window = MainWindow()
    window.show()
    sys.exit(app.exec())
```

`QApplication` — обязательный объект, запускает всё приложение. `sys.argv` — аргументы командной строки (Qt их использует для флагов платформы). `"Fusion"` — встроенный стиль Qt, хорошо выглядит на всех платформах. `QPalette` — цветовая схема приложения. `app.exec()` — запускает **цикл событий** (event loop): программа зависает здесь и обрабатывает клики, таймеры, перерисовки. `sys.exit(...)` — корректный выход с кодом возврата.

## Итог: какие концепции ООП и где

|Концепция|Где в коде|
|---|---|
|Абстрактный класс|`AbstractGameObject`, `AbstractCell`, `AbstractGameWidget`|
|Наследование|`BulbCell → EmptyCell → AbstractCell → AbstractGameObject`|
|Полиморфизм|`is_passable()`, `get_symbol()`, `draw_content()` — разное поведение, один вызов|
|Инкапсуляция|`__row`, `__state`, `__cells` с `@property`|
|Магические методы|`__str__`, `__repr__`, `__eq__`, `__len__`, `__getitem__`, `__setitem__`, `__iter__`, `__contains__`|
|Singleton|`GameManager.__new__`, `PulseAnimator.get()`|
|Observer (сигналы/слоты)|`Signal`, `connect()` в PySide6|
|Template Method|`paintEvent` → `draw_content()` в `AbstractGameWidget`|
|Генератор|`yield` в `Grid.__iter__`|
|Статические методы|`@staticmethod` в `GameLogic`, `LevelGenerator`|
#### Игра судоку
Это будет очень похоже на предядущую игру в плане кода скорее всего
- **Наследование и Абстрактные классы:** 
Можно создать абстрактный класс , от ктр наследуются класс с цифрами, и изначальными данными  и класс с пустыми клетками, куда игрок вписывает недостающие цифры
- **Инкапсуляция:** 
Класс будет скрывать двумерный массив с цифрами и хранить методы для проверки правил, чтобы цифры не повторялись в блоках 3×3, строках и столбцах
- **Специальные методы:** 
1) `__getitem__` / `__setitem__`: чтобы обращаться к клетке по координатам `board[0, 5]`
2) `__eq__`: можно сравнивать две доски (например, текущую доску игрока и доску с решением)
- **Графический интерфейс:** 
Очень удобно будет делать на кнопках или текстовых полях
