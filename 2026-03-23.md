### Comments on the data

**1. Data types** Most columns have the correct data types, but there are some problems:
- **Count of Stolen Animals** — should be integer (int), but in the real dataset it is float64. This is not a big problem, but it looks strange because you cannot steal half an animal.
- **Year and Month** — stored as string (object), for example "Jan-2010". It would be much better to have it as datetime type or at least separate columns for Year (int) and Month (int or category). String format makes it harder to sort by time or make time-based charts.
- Other columns (**Type of Animal**, **Borough**, **Type Of Offence**, **Animal Recovered**) are correctly string/object — this is fine for categorical data.

**2. Errors and problems in the data**

- **Duplicates** There are **914 duplicate rows** — this is about 16% of all 5720 rows. This means probably that the same theft case was probably entered several times. *I should remove duplicates before doing any serious analysis.
- **Missing values** From the output we see:
    - Borough - 0 missing
    - Type of Animal - 0 missing
    - Animal Recovered - 0 missing
    - Year and Month - 0 missing 
    - But **Type Of Offence** and some other columns show 0 missing in one place and later look suspicious. We need to double-check if "0" really means no missing values or if it is a mistake in the printout.
- **Outliers and strange values**
    - In **Count of Stolen Animals**:
        - minimum = 0 (strange — why report a theft if zero animals were stolen?)
        - maximum = **1000** — this looks like a possible error or very extreme case (maybe bulk theft from a farm or pet shop?)
        - mean ≈ 1.96 and median = 1.0 → most cases are 1 animal, but a few very large numbers pull the mean up → We should investigate rows where Count ≥ 10–20, because they can strongly affect averages and trends.
- **Other possible issues**
    - Many cases have **"Not Recovered"** — it would be interesting to calculate the real recovery rate (%).
    - Column names have inconsistent capital letters (**"Type Of Offence"** vs **"Type of Animal"**) — small problem, but better to make them uniform.
    - "Theft and Handling", "Robbery", "Violence Against the Person" — these categories are logical, but we should check if all values are written consistently (no spelling mistakes like "TheftandHandling" or extra spaces).

**Summary** The dataset is useful and contains interesting information about animal theft in London, but right now it has several problems:

- too many duplicates (914)
- suspicious zero and very high values in Count of Stolen Animals
- Count column should be integer, not float
- Year-Month better as datetime

Before visualisation or modelling, I would:
1. Remove duplicates
2. Check / clean outliers in Count of Stolen Animals
3. Convert Year and Month to proper date format
4. Make column names consistent
After these steps the data will be much cleaner and more reliable.
